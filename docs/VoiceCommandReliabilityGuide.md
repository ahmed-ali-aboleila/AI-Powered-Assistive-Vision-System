# Voice Command Reliability Guide

دليل سريع لتشغيل مساعد الصوت بسلاسة وثبات، خصوصا مع أوامر التعرف على الوجوه والمشاعر لمساعدة شخص كفيف.

## الهدف

المطلوب أن النظام:

- لا ينفذ أوامر من نفسه.
- لا يلتقط صوت المساعد نفسه كأنه أمر من المستخدم.
- يسمع أوامر المستخدم بشكل ثابت.
- ينفذ أوامر `face recognition` و`recognize emotion` بسرعة وبأقل أخطاء ممكنة.
- يرجع لانتظار المستخدم بعد كل أمر بدون لخبطة.

## ملخص لصاحب النظام أو أي AI يشتغل عليه

هذا الملف هو مرجع تشغيل وتشخيص. لو AI آخر سيعدل النظام عند صاحبك، لازم يفهم الآتي قبل أي تعديل:

- لا تبدأ بتغيير القاموس أو thresholds قبل تحديد نوع المشكلة: wake false trigger، command recognition، self-capture، أو vision backend.
- لا تجعل STT wake مفتوحا في الإنتاج إلا لسبب واضح ومؤقت.
- لا تجعل أوامر الرؤية الحساسة أو المتشابهة تعتمد على جمل طويلة.
- لا تضف عبارات عامة جدا إلى القاموس لأنها قد تزود الأوامر الوهمية.
- لا تحفظ raw audio أو raw utterances في logs افتراضيًا حفاظًا على الخصوصية.
- أي تغيير لازم يتم اختباره بأمر face وأمر emotion على الأقل 10 مرات.

أهم نجاح مطلوب: المستخدم الكفيف يطلب الوظيفة بجملة قصيرة، والنظام يرد بوضوح، ولا ينفذ شيء لم يطلبه المستخدم.

## طريقة عمل النظام باختصار

النظام لا ينفذ الكلام مباشرة من الميكروفون. المسار الطبيعي هو:

1. النظام يكون في وضع انتظار `standby`.
2. ينتظر كلمة تنبيه مثل `hi egb` أو `مرحبا`.
3. بعد قبول كلمة التنبيه يدخل وضع الاستماع للأمر.
4. يسجل أمر واحد لمدة محدودة.
5. يحول الصوت إلى نص.
6. يصحح أخطاء STT الشائعة ويطابق النص مع قاموس الأوامر.
7. يقيس الثقة في الأمر.
8. لو الأمر واضح ينفذه.
9. لو الأمر غير واضح يطلب إعادة أو يرفض بأمان.
10. لو الأمر حساس، مثل إغلاق النظام، يطلب تأكيد.

## طبقات العزل والحماية

النظام عنده عزل منطقي قبل تنفيذ الأمر، وليس sandbox كامل على مستوى نظام التشغيل.

معنى ذلك:

- الميكروفون لا يرسل الكلام مباشرة إلى التنفيذ.
- أي كلام لازم يمر أولا على wake.
- بعد wake، النظام يسمع أمر واحد فقط داخل وقت محدود.
- الأمر لازم يطابق قاموس الأوامر أو يتم تصحيحه لتعبير معروف.
- الأمر لازم ينجح في سياسة الثقة confidence policy.
- الأوامر الحساسة لا تنفذ مباشرة وتحتاج confirmation.
- لو الأمر غير واضح أو منخفض الثقة، النظام يطلب إعادة أو يرفض بأمان.
- لو حصل timeout أو interruption، دورة الأمر تتقفل ويرجع النظام لحالة انتظار.

لكن هذا لا يعني أن كل controller معزول من نظام التشغيل. بعد ما الأمر يعدي كل البوابات ويوصل للـ controller، فالـ controller يستطيع تشغيل الوظيفة المطلوبة مثل الكاميرا أو التعرف على الوجه. لذلك جودة العزل تعتمد على:

1. ضبط wake.
2. ضبط confidence.
3. قاموس أوامر واضح.
4. منع تسريب صوت السماعة للميكروفون.
5. جعل الأوامر الخطيرة protected وتحتاج تأكيد.

## خريطة القرار من الصوت إلى التنفيذ

```text
Microphone
  -> Wake detection
  -> Active listening
  -> Speech-to-text
  -> Command post-processing
  -> Parser and command lexicon
  -> Confidence decision
  -> Confirmation if protected
  -> Resolver
  -> Controller execution
  -> Spoken result
  -> Back to standby/listening
```

لو أي خطوة من الخطوات قبل `Controller execution` فشلت، المفروض النظام لا ينفذ الأمر.

## ملفات المشروع المهمة

لو AI سيصلح أو يضبط النظام، هذه أهم الملفات:

```text
core/assistant_runtime.py
```

يدير دورة التشغيل: standby، wake، listening، confidence، retry، refuse، speaking، والرجوع للحالة المناسبة.

```text
core/stt.py
```

يدير التقاط الصوت، timeouts، اختيار اللغة، بدائل STT، Vosk/Google/fallback، ونتائج confidence.

```text
core/wake_word.py
```

يدير wake phrases مثل `hi egb` و`مرحبا`، ويقرر هل الجملة wake صحيحة أم لا.

```text
core/command_post_processing.py
```

يصحح أخطاء STT الشائعة مثل `fish recognition` إلى `face recognition`.

```text
core/lexicon/command_lexicon.json
```

قاموس الأوامر. هنا تضيف عبارات المستخدم الحقيقية وأخطاء STT المتكررة.

```text
core/parser.py
```

يطابق النص مع intent، ويطبق thresholds حسب مستوى الخطورة.

```text
core/resolver.py
```

يربط intent بالوظيفة الفعلية، ويفرض confirmation على الأوامر المحمية.

```text
controllers/vision_controller.py
```

ينفذ face recognition وemotion recognition ويتعامل مع الكاميرا والموديلات والـ face database.

```text
settings/user_profile.json
```

إعدادات المستخدم: wake mode، confidence thresholds، max listen seconds، وسياسات fallback.

## حدود العزل التي لا يجب كسرها

لا تجعل أي مسار يتجاوز هذه البوابات:

```text
Wake -> Listen -> STT -> Post-processing -> Parser -> Confidence -> Resolver -> Controller
```

ممنوع عمليًا:

- تنفيذ controller مباشرة من نص STT خام بدون parser.
- تنفيذ أمر face/emotion من جملة wake فقط.
- تنفيذ أمر منخفض الثقة بدون retry أو رفض آمن.
- تنفيذ أمر protected بدون confirmation.
- اعتبار أي كلام أثناء صوت TTS أمرًا عاديًا بدون turn-taking أو preemption policy.

## إعدادات أمان مقترحة عند وجود أوامر وهمية

لو الجهاز عند صاحبك ينفذ أوامر من نفسه، ابدأ بهذه الإعدادات:

```json
{
  "stt_wake_allowed_in_production": false,
  "command_allow_missing_confidence_for_unprotected": false,
  "command_max_retry_cycles": 1,
  "closed_vocabulary_retry_limit": 1
}
```

لو ما زال النظام يسمع أوامر وهمية، لا توسع القاموس. أولا افحص الميكروفون وصوت السماعة ومكان الجهاز.

## أوامر الرؤية المهمة

استخدم عبارات قصيرة وثابتة. الأفضل أن يلتزم المستخدم بواحدة أو اثنتين فقط لكل وظيفة.

### التعرف على الشخص

English:

```text
hi egb
face recognition
```

أو:

```text
hi egb
recognize face
```

Arabic:

```text
مرحبا
اتعرف على الوش
```

أو:

```text
مرحبا
اعرف الوش
```

### قراءة المشاعر

English:

```text
hi egb
recognize emotion
```

أو:

```text
hi egb
read emotion
```

Arabic:

```text
مرحبا
اعرف المشاعر
```

أو:

```text
مرحبا
اقرا المود
```

## أهم قاعدة للاستخدام

لا تقل كلمة التنبيه والأمر في نفس الجملة أثناء الاختبار.

الأفضل:

```text
hi egb
انتظر رد المساعد
face recognition
```

وليس:

```text
hi egb face recognition
```

النظام مصمم أن يتعامل مع wake فقط كبداية للدورة، ثم يسمع الأمر بعد ذلك. فصل الجملتين يقلل الأخطاء جدا.

## إعدادات موصى بها لصديقك

راجع ملف:

```text
settings/user_profile.json
```

الإعدادات المفضلة للثبات:

```json
{
  "wake_primary_mode": "keyword_low_power",
  "wake_fallback_mode": "hardware_trigger",
  "stt_wake_allowed_in_production": false,
  "max_listen_seconds": 8.0,
  "command_confidence_high_threshold": 0.82,
  "command_confidence_medium_threshold": 0.55,
  "command_protected_threshold": 0.9,
  "command_max_retry_cycles": 1,
  "closed_vocabulary_retry_limit": 1
}
```

لو النظام عنده بينفذ أوامر من نفسه، خلي هذا الإعداد أكثر تحفظا:

```json
{
  "command_allow_missing_confidence_for_unprotected": false
}
```

هذا يقلل تنفيذ الأوامر عندما مزود التعرف على الصوت لا يرجع confidence واضح.

## منع النظام من سماع نفسه

أغلب مشاكل "النظام بيسمع من نفسه" تحصل بسبب أن صوت السماعة يدخل في الميكروفون.

افعل الآتي بالترتيب:

1. استخدم سماعة أذن أو سماعة بعيدة عن الميكروفون.
2. قلل صوت TTS إلى مستوى واضح لكن غير عالي.
3. ضع الميكروفون قريب من المستخدم وليس قريب من السماعة.
4. لو الجهاز لابتوب، جرب ميكروفون USB خارجي.
5. لا تختبر في غرفة بها صدى أو صوت تلفزيون.
6. اترك نصف ثانية بعد رد المساعد قبل قول الأمر.
7. لا تضع الهاتف أو السماعة أمام الميكروفون مباشرة.

## منع الأوامر الوهمية

لو النظام ينفذ أوامر بدون كلام واضح:

1. تأكد أن `stt_wake_allowed_in_production` تساوي `false`.
2. استخدم wake واضح فقط: `hi egb` أو `مرحبا`.
3. لا تجعل أي فيديو أو شخص آخر يقول عبارات قريبة من wake.
4. قلل حساسية الميكروفون من إعدادات Windows.
5. اجعل `command_allow_missing_confidence_for_unprotected` تساوي `false`.
6. استخدم أوامر قصيرة من القاموس بدل جمل طويلة.

## تشخيص المشكلة حسب العرض

### العرض: النظام يستيقظ وحده

السبب المرجح:

- صوت السماعة داخل الميكروفون.
- wake mode مضبوط على STT بشكل واسع.
- فيديو أو شخص قريب يقول عبارة شبيهة بالـ wake.
- حساسية الميكروفون عالية.

الإصلاح:

1. تأكد أن `stt_wake_allowed_in_production` = `false`.
2. قلل صوت السماعة.
3. استخدم سماعة أذن أو ميكروفون خارجي.
4. اختبر wake فقط 10 مرات بدون أوامر.
5. لا تعدل face/emotion قبل أن يثبت wake.

### العرض: النظام يسمع أمر face كأنه emotion أو العكس

السبب المرجح:

- عبارات المستخدم طويلة أو متشابهة.
- القاموس فيه عبارات عامة أكثر من اللازم.
- STT يرجع بدائل متقاربة.

الإصلاح:

1. اجعل المستخدم يستعمل عبارتين ثابتتين فقط:

```text
face recognition
recognize emotion
```

2. أضف أخطاء STT المتكررة فقط إلى `stt_mistakes`.
3. لا تضف كلمات عامة مثل `check`, `look`, `see` كأوامر مستقلة.
4. اختبر كل أمر 10 مرات.

### العرض: النظام لا يسمع المستخدم إلا أحيانا

السبب المرجح:

- الميكروفون بعيد.
- noise أو echo.
- المستخدم يبدأ الكلام قبل انتهاء صوت المساعد.
- `max_listen_seconds` قصير أو الجملة طويلة.

الإصلاح:

1. قرب الميكروفون من المستخدم.
2. اترك نصف ثانية بعد رد المساعد.
3. استخدم جملة قصيرة.
4. اجعل `max_listen_seconds` حوالي `8.0`.
5. اختبر في غرفة هادئة.

### العرض: الأمر يتعرف صح لكن الرؤية تفشل

السبب المرجح:

- الكاميرا غير متاحة.
- لا يوجد وجه واضح.
- الإضاءة ضعيفة.
- face database أو emotion model غير جاهز.

الإصلاح:

1. اختبر الكاميرا خارج النظام.
2. اختبر الإضاءة والزاوية.
3. راجع رسائل مثل `no_face_detected`.
4. راجع `controllers/vision_controller.py` وملفات مشروع الرؤية.
5. لا تعدل STT إذا كانت المشكلة في vision backend.

## تحسين سماع أوامر المستخدم

لو النظام مرة يسمع صح ومرة غلط:

1. خلي المستخدم يقول الأمر بنفس الصيغة كل مرة.
2. استخدم لغة واحدة في الأمر الواحد: عربي فقط أو إنجليزي فقط.
3. قل الأمر ببطء طبيعي، بدون مد زائد.
4. لا تبدأ الكلام قبل ما المساعد يخلص نغمة أو رد الاستعداد.
5. اختبر أمر واحد 10 مرات وسجل النتائج.
6. لو عبارة معينة تتعرف غلط باستمرار، أضفها في `stt_mistakes` أو `phrases` داخل `core/lexicon/command_lexicon.json`.

مثال لو STT يسمع `face recognition` كأنها `fish recognition`، القاموس يحتوي بالفعل على تصحيح:

```json
"stt_mistakes": [
  "run the fish recognition"
]
```

## صيغة اختبار ثابتة

استخدم هذا الاختبار قبل أي تعديل:

### اختبار wake

كرر 10 مرات:

```text
hi egb
```

المطلوب: يقبل 9 مرات على الأقل، ولا يستيقظ بدون wake.

### اختبار face recognition

كرر 10 مرات:

```text
hi egb
face recognition
```

المطلوب:

- لا ينفذ الأمر قبل wake.
- لا يخلط بين face وemotion.
- يرجع رد واضح: تم التعرف على الوجه أو لم أجد وجه واضح.

### اختبار emotion

كرر 10 مرات:

```text
hi egb
recognize emotion
```

المطلوب:

- لا يعمل emotion بدل face إلا لو المستخدم طلب emotion.
- لو مفيش وجه واضح، يقول ذلك ولا يخترع نتيجة.
- لو تم التعرف على face قبلها، يمكن استخدام نفس face id في المتابعة.

## فحص الكاميرا والرؤية

مشاكل الرؤية نفسها قد تبدو كأنها مشكلة صوت. افصل بينهما:

1. تأكد أن الكاميرا تعمل من تطبيق Camera في Windows.
2. تأكد من أن الشخص أمام الكاميرا وإضاءة الوجه جيدة.
3. تأكد أن face database موجودة ومليانة بالأشخاص المطلوبين.
4. تأكد أن موديل emotion موجود.
5. لو الرد `no_face_detected`، المشكلة غالبا كاميرا/إضاءة/زاوية وليست STT.
6. لو الرد `vision_face_dependency_missing` أو `vision_emotion_dependency_missing`، المشكلة dependencies أو ملفات موديل.

## Prompt جاهز لأي AI عند صاحبك

يمكن لصاحبك إرسال هذا النص لأي AI مع المشروع:

```text
راجع docs/VoiceCommandReliabilityGuide.md أولا. عندي مساعد صوتي لشخص كفيف يستخدم face recognition وemotion recognition. المشكلة أن النظام أحيانا يسمع أوامر من نفسه أو يسمع أوامر المستخدم بشكل غير ثابت.

لا تبدأ بتعديل واسع. شخّص المشكلة أولا: هل هي wake false trigger، self-capture من السماعة، STT command recognition، command lexicon، confidence policy، أم vision backend.

حافظ على مسار العزل:
Wake -> Listen -> STT -> Post-processing -> Parser -> Confidence -> Resolver -> Controller.

لا تجعل أي controller يتنفذ من transcript خام. لا توسع command_lexicon.json بعبارات عامة جدا. لا تحفظ raw audio أو raw utterances. اختبر face recognition وrecognize emotion عشر مرات لكل أمر بعد أي تعديل.
```

## قواعد تعديل القاموس

عند تعديل `core/lexicon/command_lexicon.json`:

- أضف العبارة التي يقولها المستخدم فعلا.
- أضف أخطاء STT المتكررة فقط في `stt_mistakes`.
- لا تضف عبارات عامة تصلح لأكثر من intent.
- حافظ على canonical قصير وواضح.
- بعد التعديل اختبر parser وruntime tests إن أمكن.

أمثلة جيدة:

```text
face recognition
recognize face
اعرف الوش
اتعرف على الوش
recognize emotion
اعرف المشاعر
```

أمثلة غير مفضلة:

```text
look
check
tell me
who is there
what is happening
```

هذه العبارات عامة وقد تسبب تنفيذ أوامر غير مقصودة.

## علامات أن المشكلة من الميكروفون

- النظام يستيقظ وهو لا يوجد شخص يتكلم.
- يسمع كلام المساعد كأنه أمر.
- يتغير النص المسموع جدا بين محاولة ومحاولة.
- المشكلة تقل عندما تستخدم سماعة أذن.

الحل: اضبط مكان الميكروفون، قلل صوت السماعة، واستخدم ميكروفون خارجي قريب من المستخدم.

## علامات أن المشكلة من قاموس الأوامر

- wake يعمل جيدا، لكن أمر معين لا يتعرف.
- نفس الخطأ يتكرر دائما في نفس العبارة.
- English يعمل لكن Arabic لا، أو العكس.

الحل: أضف العبارة كما يقولها المستخدم فعلا إلى:

```text
core/lexicon/command_lexicon.json
```

داخل intent المناسب:

- `recognize_face`
- `recognize_emotion`

وأضف أخطاء STT المتكررة في `stt_mistakes`.

## عبارات لا ننصح بها

تجنب العبارات الطويلة مثل:

```text
please can you check who is in front of me now
```

الأفضل:

```text
face recognition
```

وتجنب:

```text
look at his mood and tell me who he is
```

الأفضل:

```text
recognize emotion
```

## Checklist قبل تسليم الجهاز للمستخدم

- [ ] wake لا يعمل إلا مع `hi egb` أو `مرحبا`.
- [ ] الأمر لا ينفذ إلا بعد wake.
- [ ] صوت المساعد لا يدخل في الميكروفون.
- [ ] `face recognition` يعمل 9 من 10 مرات على الأقل.
- [ ] `recognize emotion` يعمل 9 من 10 مرات على الأقل.
- [ ] النظام لا يخلط بين face وemotion.
- [ ] عند عدم وجود وجه، النظام يقول ذلك بوضوح.
- [ ] عند عدم وضوح الأمر، النظام يطلب إعادة أو يرفض بأمان.
- [ ] `command_allow_missing_confidence_for_unprotected` مضبوط حسب مستوى الأمان المطلوب.
- [ ] الكاميرا والإضاءة وقاعدة بيانات الوجوه جاهزة.

## اختبارات قبول نهائية

قبل اعتبار النظام جاهز لصاحبك، نفذ الاختبارات التالية:

1. بدون كلام لمدة دقيقتين: النظام لا ينفذ أي أمر.
2. تشغيل صوت المساعد قرب الجهاز: النظام لا يلتقط ردوده كأوامر.
3. `hi egb` عشر مرات: wake يعمل 9 مرات على الأقل.
4. بدون wake، قل `face recognition`: لا يتم التنفيذ.
5. مع wake، قل `face recognition` عشر مرات: التنفيذ الصحيح 9 مرات على الأقل.
6. مع wake، قل `recognize emotion` عشر مرات: التنفيذ الصحيح 9 مرات على الأقل.
7. قل أمر غير واضح: النظام يطلب إعادة أو يرفض بأمان.
8. ضع وجه غير واضح أمام الكاميرا: النظام يقول إنه لا يجد وجه واضح.
9. افصل الكاميرا مؤقتا: النظام يعطي خطأ آمن ولا ينهار.
10. بعد كل أمر، النظام يرجع لحالة انتظار مناسبة ولا يظل عالقًا في listening.

## النتيجة المتوقعة بعد الضبط

بعد تطبيق هذا الدليل، السلوك المتوقع هو:

- النظام لا يستيقظ من الضوضاء العادية.
- النظام لا يسمع صوته كأمر.
- المستخدم يقول wake ثم أمر قصير.
- face recognition وemotion recognition لا يختلطان.
- الأخطاء تتحول إلى طلب إعادة أو رفض آمن، وليس تنفيذ عشوائي.
- مشاكل الكاميرا تظهر كرسائل vision واضحة، وليس كأنها مشكلة صوت.

## الخلاصة

لأفضل جودة:

1. Wake منفصل عن الأمر.
2. أوامر قصيرة وثابتة.
3. سماعة لا تتسرب للميكروفون.
4. ميكروفون قريب من المستخدم.
5. إعدادات confidence محافظة لو فيه أوامر وهمية.
6. تحديث `command_lexicon.json` بعبارات المستخدم الحقيقية.

بهذا الشكل النظام يكون أكثر سلاسة، ويقل جدا احتمال أنه ينفذ أمر من نفسه أو يسمع الأمر غلط.
